## 1. Software-Architecture Cleanup

What it is:
- Software-architecture cleanup means organizing files, responsibilities, and documentation so a project is easier to understand and extend.

Why it is used here:
- TyreVisionX had useful work, but it was spread across multiple notebook folders, report folders, and parallel training paths.
- Cleanup makes it clear what is current, what is historical, and what is planned.

Why not another approach yet:
- A full rewrite would be risky because it could break working supervised and historical baseline code.
- The safer approach is to document and scaffold first.

How it relates to TyreVisionX:
- The repo is preparing for a new anomaly-detection research track, so the structure needs to support supervised work, anomaly work, and future web curation without mixing them.

## 2. Why Repo Structure Matters in ML Projects

What it is:
- Repo structure is the way code, data configs, reports, notebooks, and logs are arranged.

Why it is used here:
- ML projects combine code, data, experiments, metrics, plots, and papers. If these are mixed together, results become hard to trust.

Why not another approach yet:
- Putting everything in one flat folder is fast at first, but it becomes confusing when experiments multiply.
- TyreVisionX is now past the single-experiment stage.

How it relates to TyreVisionX:
- The new folders separate supervised baselines, anomaly plans, web-data curation plans, historical reports, and current status reports.

## 3. Canonical Pipeline

What it is:
- A canonical pipeline is the main recommended path for running a project.
- It is the path future users should trust first.

Why it is used here:
- TyreVisionX has both a config-driven ResNet path and older SimpleCNN/frozen-feature baseline scripts.
- Calling one path canonical avoids confusion.

Why not another approach yet:
- Merging every old script into one pipeline now could break historical experiments.
- It is better to name the canonical path first, then migrate carefully later.

How it relates to TyreVisionX:
- The canonical supervised path is `configs/train/...` plus `src/train.py`, `src/evaluate.py`, and `src/data/...`.

## 4. Legacy Code

What it is:
- Legacy code is older code that may still be useful but is no longer the preferred main path.

Why it is used here:
- The Day 3 and Day 5 baseline work is valuable because it records earlier learning and comparison results.

Why not another approach yet:
- Deleting legacy code would lose useful research history.
- Force-moving everything could break imports or make old reports impossible to interpret.

How it relates to TyreVisionX:
- Files like `src/train_baseline.py`, `src/eval_baseline.py`, and `scripts/day5_seed_sweep.py` are preserved as historical/experimental work.

## 5. Anomaly Detection in This Project

What it is:
- Anomaly detection learns normal patterns and flags samples that look unusual.

Why it is used here:
- The professor's new direction is to train on good tyre images only.
- This helps when defect examples are rare, incomplete, or hard to label.

Why not another approach yet:
- A supervised classifier needs examples of both good and defect images.
- Anomaly detection can start from good-only data, but it still needs careful validation with known defects.

How it relates to TyreVisionX:
- A good-only tyre model can help flag unusual tyres and later help curate likely-good web images for human review.
