# Process Log 20260419_225006

## Pretrained Weights

What it is: learned parameters from a model trained before this project, usually on a large dataset such as ImageNet.

Why it is used here: TyreVisionX has limited tyre data, so pretrained ResNet features give a meaningful visual embedding space.

Why not another approach yet: training a feature extractor from scratch would require more data and validation.

How it is implemented: the run used `~/.cache/torch/hub/checkpoints/resnet18-f37072fd.pth`.

## Offline Model Readiness

What it is: the ability to run a model without downloading weights or data during execution.

Why it is used here: the environment is restricted and previous anomaly execution failed because weights were unavailable.

Why not another approach yet: network-dependent runs are less reproducible.

How it is implemented: `scripts/anomaly/check_anomaly_readiness.py` checks config paths, Torch caches, project artifacts, and manifests.

## Backbone Selection

What it is: choosing the CNN architecture used to extract features.

Why it is used here: ResNet18 is the configured first baseline backbone.

Why not another approach yet: changing backbones before running the baseline would make the first result harder to interpret.

How it is implemented: ResNet18 was selected because its cached torchvision weight file was found.

## Feature Extractor Vs Classifier

What it is: a feature extractor outputs embeddings; a classifier outputs class predictions.

Why it is used here: anomaly detection compares images to the normal training distribution rather than training a defect classifier.

Why not another approach yet: supervised classification already exists separately and should not be mixed into the first anomaly result.

How it is implemented: `ResNetEmbeddingExtractor` removes the final classifier layer and uses pooled features.

## Frozen Features For Anomaly Detection

What it is: using a pretrained model without updating its weights.

Why it is used here: it gives a stable baseline and avoids overfitting to a small normal set.

Why not another approach yet: fine-tuning anomaly features needs a stronger validation plan.

How it is implemented: all encoder parameters have `requires_grad=False`.

## Mahalanobis Scoring

What it is: a distance from the normal training distribution that accounts for feature covariance.

Why it is used here: it is simple, explainable, and appropriate for a first normal-only embedding baseline.

Why not another approach yet: heavier methods should be compared after this baseline is understood.

How it is implemented: normal-train embeddings fit a Mahalanobis scorer; higher score means more anomalous.

## Threshold Calibration On Validation

What it is: choosing the decision threshold using validation data only.

Why it is used here: the test set must remain a final evaluation, not a tuning set.

Why not another approach yet: tuning on test would inflate claims.

How it is implemented: threshold `12.147390651173238` was selected on validation with normal FPR <= 10%.

## Test-Once Evaluation Discipline

What it is: evaluate on test after all choices are fixed.

Why it is used here: it keeps the anomaly report honest for professor review.

Why not another approach yet: repeated test tuning would undermine the result.

How it is implemented: D1 test metrics were computed once after validation threshold selection.

## Why Tyre Anomaly Is Current Priority

What it is: focusing the research milestone on tyre-specific anomaly readiness.

Why it is used here: the project needs a credible tyre result before broader data-platform work.

Why not another approach yet: generalized platform abstractions are less valuable if the tyre baseline is not understood.

How it is implemented: this prompt updated anomaly readiness, execution, reports, and notebook rather than web scaling.

## Why Not Trust Random-Weight Results

What it is: a random-weight model has no learned visual features.

Why it matters here: anomaly scores from random embeddings would not reflect tyre normality or defects.

Why not another approach yet: smoke tests can use random weights only to test software paths, not research quality.

How it is implemented: the run used cached pretrained ResNet18 weights and reports avoid random-weight claims.

## Local Checkpoint Reuse

What it is: reusing a project checkpoint as a feature source.

Why it is considered here: historical TyreVisionX checkpoints exist locally.

Why not use it yet: the cached official ResNet18 weight was available, and supervised checkpoint compatibility needs separate validation.

How it is implemented: readiness reports list local checkpoints but mark them not selected automatically.

## Professor-Discussable Baseline

What it is: a baseline with clear data split, method, threshold policy, metrics, and limitations.

Why it is used here: it creates a concrete research artifact for discussion.

Why not claim more yet: recall is low and false negatives are high.

How it is implemented: reports show AUROC, AUPRC, confusion matrix, false negatives, false positives, score plots, and next steps.
