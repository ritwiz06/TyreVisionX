## Source Acquisition Layer

What it is:
Source acquisition is the controlled process of finding candidate image sources before downloading, filtering, or reviewing them.

Why it is used here:
TyreVisionX needs more likely-normal tyre images later, but those images must have provenance and review records.

Why not another approach yet:
Bulk scraping would be fast but unsafe, hard to audit, and likely to violate provider rules.

How it is implemented in TyreVisionX:
The project now has `configs/web_collection/provider_sources.yaml`, source-specific provider scaffolds, manual Google CSV import, and source-acquisition docs.

## Provider Abstraction

What it is:
A provider abstraction is a common way to represent different sources such as Wikimedia, Pexels, Unsplash, Flickr, internal data, or manual Google discovery.

Why it is used here:
Different sources have different credentials, metadata, and rights fields, but TyreVisionX needs one candidate metadata format.

Why not another approach yet:
Hardcoding one provider would make the research workflow brittle and difficult to compare.

How it is implemented in TyreVisionX:
Provider modules under `src/web_collection/` validate configuration and expose clear status. They do not hide missing credentials.

## Google Manual Discovery

What it is:
Google manual discovery means a researcher uses a browser, manually approves candidate rows, and enters them into a CSV.

Why it is used here:
It allows discovery without scraping Google HTML or bypassing provider terms.

Why not another approach yet:
Automated Google scraping is explicitly forbidden, brittle, and not appropriate for a research audit trail.

How it is implemented in TyreVisionX:
`src/web_collection/providers_manual_google.py` and `scripts/web_collection/import_manual_google_discovery.py` read a researcher-prepared CSV only.

## Source and License Metadata

What it is:
Source and license metadata record where an image came from, what page describes it, and what usage terms or attribution text were available.

Why it is used here:
Web candidates may later be discussed in research or productization settings, so provenance must not be lost.

Why not another approach yet:
Saving only image files would make later rights review and reproducibility much weaker.

How it is implemented in TyreVisionX:
Candidate records now include fields such as `provider`, `query_id`, `license_name`, `license_url`, `attribution_text`, `view_type`, and `notes`.

## Candidate Versus Label

What it is:
A candidate is an image that may be useful after review. A label is a trusted training target.

Why it is used here:
Search results can be irrelevant, damaged, mislabeled, duplicated, or low quality.

Why not another approach yet:
Auto-labeling web images as normal would contaminate a good-only anomaly dataset.

How it is implemented in TyreVisionX:
Docs and scripts keep all web images as research candidates until a human review status such as `approved_likely_normal` is assigned.

## API Credentials

What it is:
API credentials are provider-issued keys required by services such as Pexels, Unsplash, and Flickr.

Why it is used here:
Official APIs are safer and more stable than page scraping.

Why not another approach yet:
The repository should not contain private keys, and collection should fail clearly when keys are absent.

How it is implemented in TyreVisionX:
Provider adapters check environment variables such as `PEXELS_API_KEY`, `UNSPLASH_ACCESS_KEY`, and `FLICKR_API_KEY` and raise helpful errors when missing.

## Wikimedia Commons

What it is:
Wikimedia Commons is a public media repository with a documented API and license metadata.

Why it is used here:
It can provide source and license fields without requiring a private API key.

Why not another approach yet:
It may be noisy for tyre inspection images, so it should be treated as candidate discovery, not trusted labels.

How it is implemented in TyreVisionX:
`src/web_collection/providers_wikimedia.py` builds official API request URLs and reports `ready_no_api_key`.

## Pilot Before Scale

What it is:
A pilot is a small controlled run before collecting thousands of images.

Why it is used here:
TyreVisionX needs to prove metadata, filtering, review queues, and promotion rules on 30-50 candidates first.

Why not another approach yet:
Scaling before review quality is understood can create a large, messy dataset that is expensive to clean.

How it is implemented in TyreVisionX:
`reports/web_collection/pilot_source_plan.md` defines the first 30-50 candidate plan and keeps bulk downloads out of this prompt.

## Review-Gated Data Growth

What it is:
Review-gated growth means candidates enter training manifests only after explicit human review.

Why it is used here:
The current anomaly baseline is weak for recall-critical inspection, so model scores cannot replace review.

Why not another approach yet:
Pseudo-labeling web data now would risk teaching the anomaly model that damaged or irrelevant tyres are normal.

How it is implemented in TyreVisionX:
The source layer feeds the existing filtering, review queue, review pack, and curated-normal promotion path.
